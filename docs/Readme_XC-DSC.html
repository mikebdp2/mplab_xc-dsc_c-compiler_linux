  1. Overview

    2. The XC-DSC language tools consist of a compiler (xc-dsc-gcc.exe), an assembler (xc-dsc-as.exe), a linker (xc-dsc-ld.exe), and an archiver/ librarian (xc-dsc-ar.exe). Additional tools distributed with this release include a binary to Intel Hex converter (xc-dsc-bin2hex.exe) and other utilities.

    The XC-DSC language tool official names are:

    • MPLAB XC-DSC C Compiler
    • MPLAB XC-DSC Assembler
    • MPLAB XC-DSC Object Linker
    • MPLAB XC-DSC Object Archiver/Librarian

    As described in the user's guides, all of the language tools are sensitive to case on the command-line, including the case of file names. In particular, the C compiler shell (xc-dsc-gcc) requires that C source files be named on the command-line with a lower-case .c extension. If an upper-case .C filename extension is used, the compiler assumes that the file is a C++ file, which is not supported.

    If this release of the Microchip MPLAB XC-DSC C compiler is a functional safety compiler, this compiler requires a functional safety license in order to operate. Once activated, all optimization levels and compiler features are available.

  8. What's New in this Release.

    9. Follows is a summary of new features with each release. Please be sure to check for migration notes or new documentation in the relevant sections of this file.

    v3.00

    MPLAB XC-DSC v3.00 is a continuation of MPLAB XC16 with focussed support for Microchip Digital Signal Controller devices (DSC). The name change may be the most visible aspect of this update, but there are plenty of changes under the hood.

    MPLAB XC-DSC v3.00 updates the GCC source base version to GCC v8.3.1. This improves our baseline support and provides access to advanced features.

    MPLAB XC-DSC v3.00 updates our binary utilities source base to Binutils v2.32.

    As bug tracking / feature tracking migrates from an XC16 database to an XC-DSC database, you may see references to both types of issue. Do not be alarmed. :)

  9. Migration Issues

    10. Migrating to XC-DSC Version 3.00

    • The installer default installation directory has been updated to reflect the product branding change; the new path is OS-root-path/xc-dsc/version. The root executable names are prefixed with xc-dsc.

    • Manifest constants defined by the compiler will take on a new form; the former __XC_ and __C30_ prefixes of constants still exist in order to maintain code compatibility, however, new macros will take the new prefix __XC_DSC_ for all new and current constants. For example:

      • __XC16_VERSION__ can be referred to by the new name __XC_DSC_VERSION__.

    • The assembler macro processor is more restrictive in the syntax that can be used with macro arguments. For example, the definition of a macro called config in previous versions of assembler include files included this statement which now results in an error:

       .pushsection \REG.sec, code

      An extra string must be added to separate the macro argument REG from the following text. As shown in this example, the "dot" in .sec must be separated from the macro name REG using the extra string \():

       .pushsection \REG\().sec, code

    • Previous version of the assembler would allow .equiv statements to be modified with a .set or .equ directive. The current assembler restricts this behaviour in line with the intended purpose of .equiv. If a symbol may be redefined, use .equ instead of .equiv to define it.

    • Previous versions of the linker would accept a linker command file (aka, linker script) that was simply included on the command line with other input files. The new linker requires the -T option or --script option to explicitly specify a linker script. If a linker script is detected without one of those options, the file will be accepted and a warning will be issued.

    • Previous versions of the linker would always include symbols defined in a linker script as global symbols in the linked executable. The new linker will include such symbols as 'local'.

    • The Compiler option -falign-arrays is no longer supported; the compiler will produce an error message if this is used.

  12. Limitations

    13. This section describes limitations of the product, and applicable workarounds when possible. For further assistance, contact Customer Support.

    The current limitations are as follows:

      Assembler

    • Arguments to the address() section attribute must be even.
    • The default section alignment is 2**1. Therefore the minimum section length is 2 bytes in data memory, and 2 PC units (3 bytes) in program memory.
    • BIN30-8/21797 - The .align directive changes section alignment regardless of max-skip.

      The max-skip parameter does have the desired effect: the location counter is not adjusted, and pad bytes are not inserted. However, the section alignment is changed.

      This example will demonstrate:

      .data
.byte 1
.align 8,0x99,2
.byte 2

      • Compiler

    • XCDSC-4 - Not all GCC options are available for use; please see the current documentation for acceptable. In particular, the new option -fsanitize=
    • XCDSC-6 - __builin_flim returns incorrect values; use inline assembly or a assembly language function
    • XC16-229 - Variables qualified with the __pack_upper_byte keyword are shown at a different address in the watch window of MPLAB IDE and the map file.

      The debugger is not able to convert the virtual address of these symbols into the real address. For more information, please refere to section 11.9 PACKING DATA STORED IN FLASH in the compiler manual.

    • Structures must be less than 4096 words in size.
    • Blocks of code (such as a conditional statement or loop) may not exceed 32K program words.
    • Procedural Abstraction limitations:

      When the procedural abstraction optimization is enabled (using the -mpa command-line option) inline assembly code must be limited to valid machine instructions.

      Invalid machine instructions, instruction sequences, or assembler directives (sectioning directives, macros, include files, etc.) must not be used. Invalid use will cause procedural abstraction compilation stage to fail, inhibiting the creation of an output file.

    • Data placed in space(ymemory) with the section directives or through legacy section(".ybss") or section(".ydata") attributes may cause a link error.
    • C30-11/28301 - No error for unsupported nested functions
    • C30-59/26772 - With -g (generate debug info), compiler-generated assembly for noload function switches back to .text before code for function
    • C30-68/27206 - DWARF line table is missing entries for two level abstracted code.
    • C30-172 - pa tool expects only labels to start at the beginning of a line (column 0)
    • C30-398 - 24-bit pointer types are not supported in debug format
    • C30-981 - Assignment of literal value to EDS pointer loads incorrect address

      • Linker

    • BIN30-178 - Linker does not validate object signatures across input files.

      The linker will not detect a mis-match of the -mlarge-arrays option among input files. As a work-around, place objects into a library before adding them to the project.

    • BIN30-179 - Linker does not support initialized data sections larger than 64K.
